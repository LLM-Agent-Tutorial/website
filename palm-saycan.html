PaLM–SayCan: A Clear, Grounded Agent Pattern

Paper ↗ GitHub ↗

Embodied agents must turn open-ended language into safe, effective behavior amid partial observability, long horizons, hardware limits, and real-time uncertainty. PaLM-SayCan addresses these issues by separating high-level reasoning (Say) from low-level feasibility (Can): it asks what is useful to do next and what the robot can actually do now, executing only actions that are both useful and feasible.

Next we break down how PaLM-SayCan separates planning (“Say”), a structured skill/tool schema, and reflective feasibility checks (“Can”) to keep robots grounded.

Planning (Say)

The LLM acts as a semantic planner over a fixed skill set. It does not invent new primitives; it scores each provided skill by usefulness while optionally revealing short reasoning for interpretability.

Example of planning (Say)

INPUT: Role: Semantic planner for a mobile manipulator. Instruction: I spilled my drink. Available Skills: ["find a sponge", "pick up sponge", "find an apple"]. Task progress: No cleanup tools found yet. Constraints: score every listed skill with a probability.

OUTPUT: Utility scores — find sponge: 0.85; pick up sponge: 0.10; find apple: 0.05. Rationale: Locate a cleaning tool before manipulation; ignore irrelevant goals.

Skill schema

SayCan treats low-level abilities as tools that must be grounded in two ways: text-to-skill grounding with human-readable names/descriptions, and execution grounding via executors with preconditions and postconditions. New skills are registered with their schema plus an affordance estimator to become available immediately.

Example of skill schema

Name: open_refrigerator
Description: Open the refrigerator door using the handle.
Preconditions: handle visible and within reach; gripper free
Postconditions: door angle > 20°
Executor: grasp → pull → release
Affordance: A_open_refrigerator(x) predicts success from detections, depth, reachability.

Reflection (Can)

Reflection is the feasibility check that keeps the plan honest. For each available skill, a pre-trained affordance function estimates the likelihood of success in the current scene. The system multiplies the Say usefulness and Can feasibility scores before executing the top skill.

Example of reflection (Can) and decision

INPUT: Instruction: I'd like a cold apple. Scene: robot is far from a closed refrigerator; handle visible but out of reach. Skills: [navigate to refrigerator, open refrigerator, pick up apple]. Planning (Say) utilities: navigate 0.50; open 0.40; pick up 0.10. Affordance estimates: navigate 0.95; open 0.20; pick up 0.05.

OUTPUT: Combined scores favor navigate to refrigerator. After moving, the affordance for open refrigerator rises, so the next step becomes opening the door, followed by picking up the apple.
